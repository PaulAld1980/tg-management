# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

**Конструктор Telegram-бота технической поддержки**  
Support Bot Constructor  
Версия 1.0 | Март 2026

---

## 1. Обзор проекта

### 1.1 Назначение

Платформа-конструктор, позволяющая владельцам бизнеса без навыков программирования создавать и настраивать Telegram-ботов технической поддержки для своих клиентов. Каждый зарегистрированный пользователь платформы управляет собственным ботом через веб-админку.

### 1.2 Технологический стек

| Компонент | Технология | Назначение |
|---|---|---|
| Telegram-бот | Python 3.12, aiogram 3.x | Взаимодействие с конечными пользователями |
| Backend / API | Django 5.x, Django REST Framework | Бизнес-логика, REST API, авторизация |
| Frontend (админка) | Vue.js 3, Composition API | Веб-интерфейс управления ботом |
| База данных | PostgreSQL 16 | Основное хранилище данных |
| Медиа-хранилище | Django media / S3-совместимое | Хранение изображений товаров |
| Кеш / Брокер | Redis 7 | Кеширование, очередь задач Celery |
| Очередь задач | Celery | Асинхронные задачи (запуск/остановка ботов) |
| Контейнеризация | Docker, Docker Compose | Развертывание и изоляция сервисов |

### 1.3 Роли пользователей

| Роль | Описание | Способ создания |
|---|---|---|
| Суперадмин | Администратор платформы, создает аккаунты владельцев | `manage.py createsuperuser` |
| Владелец бота | Бизнес-пользователь, конфигурирует своего бота | Создается суперадмином |
| Конечный пользователь | Клиент бизнеса, взаимодействует с ботом в Telegram | Не имеет аккаунта в системе |

## 2. Функциональные требования

### 2.1 Аутентификация и настройка бота

#### 2.1.1 Авторизация

- Вход в Vue-админку по логину и паролю (JWT-аутентификация через DRF)
- Один аккаунт = один Telegram-бот
- Аккаунты создаются только суперадмином (через Django Admin или CLI)

#### 2.1.2 Первичная конфигурация

- При первом входе владелец обязан задать Bot Token (полученный от @BotFather)
- До ввода валидного токена доступ к остальным разделам админки заблокирован
- Система валидирует токен через Telegram Bot API (`getMe`)
- После валидации токен сохраняется в зашифрованном виде

### 2.2 Конструктор кнопок (дерево навигации)

#### 2.2.1 Концепция

Владелец бота строит древовидную структуру навигации. Каждый узел дерева представляет собой кнопку в интерфейсе Telegram-бота. Узел может быть одного из двух типов:

| Тип узла | Поведение в боте | Дочерние узлы |
|---|---|---|
| Категория (`category`) | При нажатии показывает inline-кнопки дочерних узлов | Да, 1 или более |
| Ответ (`answer`) | При нажатии отправляет заготовленный текстовый ответ | Нет (листовой узел) |

#### 2.2.2 Поля узла

| Поле | Тип | Описание |
|---|---|---|
| `label` | string (max 64) | Текст кнопки, отображаемый пользователю |
| `node_type` | enum: `category` \| `answer` | Тип узла |
| `parent` | FK -> self (nullable) | Родительский узел (null = корневой) |
| `answer_text` | text (nullable) | Текст ответа (только для типа `answer`) |
| `order` | integer | Порядок отображения среди siblings |

#### 2.2.3 Ограничения и правила

- Глубина вложенности не ограничена (рекомендация: до 5 уровней)
- Узел типа `answer` не может иметь дочерних узлов
- Узел типа `category` должен иметь хотя бы одного ребенка
- Корневые узлы отображаются на главном экране бота
- Drag-and-drop для изменения порядка и перемещения узлов во Vue-админке

### 2.3 Связь с менеджером

#### 2.3.1 Конфигурация (админка)

- Владелец создает специальную корневую кнопку с типом `contact_manager`
- В настройках указывает Telegram Chat ID менеджера (или нескольких)
- Текст приветственного сообщения настраивается в админке

#### 2.3.2 Пользовательский сценарий (Telegram)

1. Конечный пользователь нажимает кнопку связи с менеджером
2. Бот предлагает описать проблему текстом (опционально прикрепить до 5 фотографий)
3. После отправки бот пересылает сообщение менеджеру с указанием данных пользователя (`username`, `user_id`)
4. Менеджер отвечает через бота (бот проксирует ответ обратно пользователю)
5. Диалог продолжается до завершения одной из сторон

### 2.4 Консультация по товарам

#### 2.4.1 Карточка товара (админка)

Владелец заполняет каталог товаров. Каждая карточка содержит:

| Поле | Тип | Обязательное |
|---|---|---|
| `name` | string (max 200) | Да |
| `description` | text | Да |
| `images` | ImageField (до 5 шт.) | Нет |
| `is_popular` | boolean | Нет (для топ-5 на главной) |
| `order` | integer | Нет |

#### 2.4.2 FAQ товара

К каждому товару привязывается набор вопросов-ответов:

| Поле | Тип | Описание |
|---|---|---|
| `product` | FK -> Product | Связь с товаром |
| `question` | string (max 256) | Текст кнопки-вопроса |
| `answer` | text | Заготовленный ответ |
| `order` | integer | Порядок отображения |

#### 2.4.3 Пользовательский сценарий (Telegram)

- Пользователь нажимает кнопку "Консультация по товару"
- Бот показывает до 5 популярных товаров (inline-кнопки) + предлагает ввести название вручную
- При ручном вводе: поиск по полю `name` (`icontains`), показ результатов
- При выборе товара: отправка карточки (фото-альбом + описание)
- Под карточкой: inline-кнопки FAQ данного товара
- При нажатии на FAQ-кнопку: бот отправляет соответствующий ответ

## 3. Модели данных

### 3.1 Схема основных моделей

| Модель | Ключевые поля | Связи |
|---|---|---|
| `BotConfig` | `token` (encrypted), `welcome_message`, `is_active` | OneToOne -> User |
| `NavigationNode` | `label`, `node_type`, `answer_text`, `order` | FK -> self (parent), FK -> BotConfig |
| `ContactManagerConfig` | `manager_chat_ids` (JSONField), `greeting_text` | OneToOne -> BotConfig |
| `Product` | `name`, `description`, `is_popular`, `order` | FK -> BotConfig |
| `ProductImage` | `image` (ImageField), `order` | FK -> Product |
| `ProductFAQ` | `question`, `answer`, `order` | FK -> Product |
| `SupportDialog` | `user_chat_id`, `manager_chat_id`, `status`, `created_at` | FK -> BotConfig |
| `DialogMessage` | `sender_type`, `text`, `photo`, `created_at` | FK -> SupportDialog |

### 3.2 Важные замечания по моделям

- `BotConfig.token` хранится в зашифрованном виде (`django-encrypted-model-fields` или Fernet)
- `NavigationNode` реализуется через MPTT или `django-treebeard` для эффективной работы с деревьями
- `ContactManagerConfig.manager_chat_ids`: JSONField со списком `chat_id` менеджеров
- `SupportDialog.status`: enum (`active`, `closed`) для управления состоянием диалога

## 5. Архитектура Telegram-бота (aiogram)

### 5.1 Принципы

- Каждый бот запускается как отдельный asyncio-процесс (управляется Celery worker)
- Бот при старте загружает конфигурацию и дерево навигации из БД через Django ORM
- При изменении конфигурации в админке: webhook/signal для hot-reload или restart
- Используется polling (long polling) для простоты; webhook опционально

### 5.2 Структура хэндлеров

| Хэндлер | Триггер | Действие |
|---|---|---|
| `start_handler` | `/start` | Приветственное сообщение + корневые кнопки |
| `navigation_handler` | Callback от inline-кнопок | Навигация по дереву / показ ответа |
| `product_search_handler` | Текстовый ввод (в режиме поиска) | Поиск товара, показ карточки |
| `product_faq_handler` | Callback от FAQ-кнопок | Показ ответа на FAQ |
| `contact_start_handler` | Кнопка связи с менеджером | Начало диалога, FSM |
| `contact_message_handler` | Сообщение в режиме диалога | Проксирование сообщения менеджеру/пользователю |

### 5.3 FSM (Finite State Machine)

Aiogram FSM используется для управления состояниями:

- `idle` — начальное состояние, навигация по кнопкам
- `product_search` — ожидание ввода названия товара
- `contact_dialog` — режим переписки с менеджером

## 7. Нефункциональные требования

### 7.1 Безопасность

- JWT-аутентификация с коротким access-токеном (15 мин) и refresh (7 дней)
- CORS-политика: только домен фронтенда
- Bot Token шифруется при хранении в БД
- Все эндпоинты требуют авторизации (кроме `/login/`)
- Данные каждого владельца изолированы (queryset-фильтрация по `request.user`)
- Rate limiting на API (throttling DRF)
- CSRF-защита для cookie-сессий (если используются)

### 7.2 Архитектура кода

- Принцип Fat Services / Skinny Views: бизнес-логика выносится в слой сервисов
- DRY: переиспользование сериализаторов, mixins, permissions
- Type hints для всех функций и методов (Python 3.10+)
- PEP 8, isort, black для форматирования
- Django ORM: `select_related` / `prefetch_related` для предотвращения N+1

### 7.3 Тестирование

- pytest + pytest-django для unit/integration тестов
- Минимальное покрытие: модели, сервисы, API-эндпоинты
- Factory Boy для генерации тестовых данных

### 7.4 Развертывание

- Docker Compose: `django`, `postgres`, `redis`, `celery-worker`, `celery-beat`, `vue` (nginx)
- Переменные окружения через `.env` (django-environ)
- Nginx как reverse proxy + раздача статики

## 8. Этапы разработки (из ТЗ v1.0)

> Рекомендуемая последовательность, при которой каждый следующий этап основывается на результатах предыдущего.

### Этап 1. Инфраструктура и скелет проекта

- Инициализация Django-проекта с модульной структурой приложений
- Docker Compose: postgres, redis, django, celery
- Конфигурация settings (split на base/dev/prod), `.env`
- Инициализация Vue.js проекта (Vite + Vue Router + Pinia)
- CI: линтеры (black, isort, flake8, eslint)

### Этап 2. Аутентификация

- Модель User (или расширение AbstractUser) + BotConfig
- DRF: JWT эндпоинты (login, refresh, me)
- Vue: страница логина, Pinia auth store, axios interceptors
- Middleware: redirect на `/settings` если токен бота не настроен

### Этап 3. Конфигурация бота

- API: CRUD для BotConfig (ввод и валидация Bot Token)
- Vue: страница `/settings` с формой токена и статусом бота
- Шифрование токена при сохранении
- Celery task: запуск/остановка aiogram-бота

### Этап 4. Конструктор навигации (дерево кнопок)

- Модель `NavigationNode` (self-referential FK, MPTT/treebeard)
- API: CRUD + move endpoint (nested serializer для GET)
- Vue: визуальный редактор дерева с drag-and-drop
- Aiogram: хэндлер навигации по дереву (динамические inline-кнопки)
- Тесты: создание дерева, навигация, валидация (answer без детей, category без ответа)

### Этап 5. Связь с менеджером

- Модели: `ContactManagerConfig`, `SupportDialog`, `DialogMessage`
- API: настройки менеджера, история диалогов
- Aiogram: FSM для режима диалога, проксирование сообщений
- Vue: страница настроек менеджера + просмотр истории
- Поддержка фотографий в сообщениях (до 5 шт.)

### Этап 6. Каталог товаров и FAQ

- Модели: `Product`, `ProductImage`, `ProductFAQ`
- API: CRUD товаров, загрузка изображений, CRUD FAQ
- Vue: страница каталога + детальная страница товара с FAQ и изображениями
- Aiogram: поиск товара (`icontains`), карточка с фото-альбомом, FAQ-кнопки
- Тесты: поиск, отображение карточки, FAQ

### Этап 7. Интеграция, тестирование, деплой

- End-to-end тестирование всех сценариев
- Нагрузочное тестирование (несколько ботов одновременно)
- Документация API (drf-spectacular / Swagger)
- Финальная конфигурация Docker Compose (production)
- Nginx конфигурация, SSL, бэкапы PostgreSQL
- Мониторинг: Sentry для ошибок, логирование

## 🛣 Roadmap (упрощённая версия)

> Разработку лучше вести от ядра (БД) к визуальной части, а бота делать параллельно или после готовности API.

### Этап 1: Проектирование БД и Backend ядро (Django + DRF)

1. Развертывание базового проекта Django.
2. Моделирование базы данных:
   - Таблица профиля клиента (связь 1к1 с `User`, хранение `bot_token`, `manager_chat_id`)
   - Таблица элементов меню (паттерн Adjacency List: ссылка `parent_id` на саму себя для бесконечной вложенности, атрибуты: название, тип узла, текст ответа)
   - Таблица Товаров (название, описание, boolean флаг "популярный")
   - Таблица Фотографий товара (связь ForeignKey с Товаром)
   - Таблица FAQ товара (связь ForeignKey с Товаром, вопрос, ответ)
3. Создание сериализаторов (Serializers) DRF.
4. Написание REST API эндпоинтов (ViewSets), настройка разрешений (Permissions), чтобы запросы фильтровались по `request.user`.

### Этап 2: Frontend интерфейс администратора (Vue.js)

1. Инициализация проекта Vue.js, настройка Vue Router, стейт-менеджера (Pinia) и библиотеки компонентов (например, Vuetify или Tailwind).
2. Реализация страницы логина и настройка работы с JWT-токенами (или сессиями DRF).
3. Создание механизма "Глобального Guard-а роутов" — если у юзера пустой `bot_token`, принудительный редирект на экран конфигурации.
4. Разработка UI интерфейса управления ботом:
   - Конструктор иерархического меню кнопок
   - Раздел настройки менеджера (`chat_id`)
   - Раздел CRUD для товаров, загрузка картинок, выбор популярных, добавление FAQ к карточке
5. Связывание Frontend с Backend API (использование Axios/Fetch).

### Этап 3: Ядро менеджера ботов (aiogram + DRF)

1. Проектирование архитектуры запуска aiogram. Т.к. ботов много (под каждый аккаунт свой токен), потребуется написать мультиботовую архитектуру (например, использование пула запущенных асинхронных тасок поллинга или единый webhook-сервер, роутящий запросы на основе токена в URL).
2. Настройка подключения aiogram к моделям БД (с помощью асинхронной обертки Django ORM `sync_to_async` или прямых запросов).

### Этап 4: Разработка логики Telegram Бота

1. Реализация динамического создания клавиатур (ReplyKeyboard) на основе данных из таблицы элементов меню.
2. Реализация FSM (Машины состояний) для логики "Написать менеджеру" (перехват сообщений в стейт и пересылка).
3. Реализация модуля товаров:
   - Генерация InlineKeyboard с 5 популярными товарами
   - Текстовый поиск через ORM (`icontains`)
   - Формирование карточки (отправка `MediaGroup` для фото)
   - Регистрация динамических Callback-хэндлеров для кнопок FAQ товара
